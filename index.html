再來說說CompoundJS

Author:Tengfei Date:2013-11-05 Categray: NodeJS

昨天又想來研究一下CompoundJS的源碼，Google了一下文檔，發現國內還是不怎麼有人玩這個，結果找到了一篇自己四個月前的文章。可惜原來的的域名不想續費了。都有三個域名，一個都不升值。拿在手裡真是浪費了。（爲什麽每次我寫文字都要扯到旁邊去，好像不扯到旁邊就不是我的風格）咦。。。這就拉回來

嗯，正文開始了。其實在我之前的文章裡面也說過，CompoundJS其實就是個Grails的一比一Node版本。她的官方文檔在這裡。安裝部署非常簡單，分分鐘的事情。其實無論後端MVC還是前端MVC最最最關鍵的還是Controller這一層。不過對於後端同學而言，Model這一層也是至關重要的，在Grails裡面這個叫Domain，其實都是一個意思，當然她們都是Rails系的。Compound之餘NodeJS猶如Django之餘Python，Cake之餘PHP，ROR之餘Ruby，Grails之餘Java(Groovy)。

要想仔細深入的玩一下這個東東，必須深入研究一下ORM的各種關係，一對一，一對多，多對多。這個，CompoundJS用的ORM就是原作本人寫的Jugglingdb,你可以把她理解為一個NodeJS版本的Hibernate。這個對於玩Java的同學再熟悉不過了。這個ORM還極為不成熟，之前直接在Express中使用的時候Bug諸多，誰用誰小心^ ^。不過最重要的是學會怎麼構架一個簡單的系統。怎麼實現一對一，一對多，多對多的關係。什麽時候應該用什麽模式。這才是最最關鍵的，數據庫層出不窮，如果一味的陷入到各種數據庫的使用中去就會陷入死衚衕，當然這是從一個前端的角度來說的，對於DBA的同學當然要精通各種DBMS的使用。還有一個就是Controller，這個在CompoundJS中也是極為關鍵的。So①Model（Jugglingdb）的使用和②Controller的使用是CompoundJS中最重要的兩個部份。這就像是書法裡面的用筆和結字不可有任何的偏廢。如果用筆不行，結字再怎麼美觀都是徒有其表，如果用筆非常有功力，結字亂了那也是非常失敗的。同樣的在構建Model的時候經過嚴密的設計，之後的Controller就可以遊刃有餘了。

書法普及文（一）

Author:Tengfei Date:2013-11-02 Categray: 書法

今天像吃了鸡血一样亢奋，刚刚随笔记录了一点感受，也不能称之为博文的，本来也就没想当什么博文的，像马磊同学说的就是生活的日志而已，生活就应该记录下来，So，我想写一篇文章，来普及一下书法。

由同事的话说开来“我小时候也有一位老师给我们教书法，那时候没好好学，也不是很喜欢，现在发现写字好也非常。。。。”虽然不是悔不当初，但是同事的语气中透漏着小小的遗憾。姑且我先入为主的这么认为吧。

你所认识的书法是错误的，完全的错误，不信听完细细道来。你所遗憾的和书法没关系，只是现在写出来的字不是太好看而已。OK，这是你的需求，我可以用半个小时解决你的这个需求。你所认为的好就是字结构的好，大家都认为结构美观，你说你练了20年30年甚至50年，我一天都没练过。可是我写两个字“人”“大”，会有多大的区别，都是一撇一捺，多一横而已。区别及其的小，至少在你看来。那为什么就不好看呢？其实书法最基础的部分是有章可循的，这其中的“法”就是结构，也称结体。好像还在卖关子似的。看这里黄自元间架结构九十二法,这里面记录了最基础的字形的结构排版的规则，只要遵循这些规则，我保证不出30分钟你就可以知道其中的奥妙。

本来想长篇大论的写一篇普及文章，以装出一幅我功力高深的样子，可是寥寥几十个字发现已经写完了。那我就弱弱的举个例子，相信聪明的你一定能够一下子看明白。这个例子在那本书里特别详尽，感兴趣的同学一定要花一点时间学习一下。保证不超过30分钟就能见到效果。

比如我要写个“和”字，这样左边笔画多，右边笔画少的左右结构的字，你可以尝试右边的“口”高低不一样的显示效果，你分别写两个字，一个“口”低一点，另一个“口”高一点。聪明的你一下子就明白了，为什么别人写的这个字就比较好看呢。因为你平常不会去关注这么细节的东西。所以有这样一条规律。左右结构的字，右边笔画少的不嫌低。左边笔画少的，左边不嫌高，如“呼”。

后记：这是从形的角度而言的，事实上这些东西会了连书法的入门都算不上。所以我一开始就说你所认识的书法是错误的。因为在我看来书法是由“形”和“质”构成的。前面只是“形”的皮毛而已。当然略花时间，这个“形”的问题就很好解决。

话说回来，事实上这个和设计里面的图形基本元素组成也有很大的关系，和中国画里的构图方式也有关系，也包括到摄影里面的构图布光。大道至简。图都是点线面构成，好的设计都遵循一定的图元（这是计算机图形学里的概念，暂借这个词来一用），圆，三角，梯形，四边形，五边形。。。这些规则在构图技巧几乎是放之四海而皆准的。当然这个在书法字形构图里面也是适用的，虽然古人肯定不会这么讲，可是自古人类的审美认知都是这样的。几千年前美的东西，今人看了也觉得很美。并且拿今人的理论去分析古人的作品惊人的一致。大象无形，大美不言。

貌似到后面又跑题了，关于书法更多的东西后面会有更为详细的文章介绍细节，上文所述的能基本理解，就算是入门四成了。^ ^

讀《交互設計指南》有感

Author:Tengfei Date:2013-11-02 Categray: 設計

昨天我又把Themeforest的Theme Category有一半的部份看了一遍，試圖從中找出普適規律，我想這種規律是有的，早在去年冬天的某天我就一直在考慮這樣一個問題。UI的設計，一定有一個通用的解決方案，模塊化的設計。這個有點像書法裡面的筆法，只要你真正掌握了用筆的方法，任憑字的結構有千變萬化，都可以遊刃有餘。事實上BootStrap就扮演了這樣一種角色，讓後端的同學對前端對設計再也不怕了。

看了很多的Theme之後，又去重新複習了一下PhotoShop的常用技巧之類的。其實技巧這種東西本不應該大力提倡的，技巧的東西都不是最本源的東西。只有把最本源的東西掌握了，就什麽也不怕了。So我又跑到西喬的部落格中去尋覓設計書列表，看到點石成金,其實就是Don’t me Think。不過在Google的時候發現了另一本更好的書，那就是本文要重點說的《交互設計指南》。本來以為是一本技巧性的書，讀過之後發現是一本哲學書，一本講述產品設計，交互設計的方法論的書。

最讓我印象深刻的是第二章，設計的四個方法論①CDC設計原則②ACD設計原則③SD設計原則，④GD設計原則。和第二章設計策略。其實本來大家都在使用這種方法了，只是沒有這麼明確的總結出來，推廣到一般。

雖然說是UI設計原則，可是第二章明明是在論述產品設計原則，設計策略中說的差異化策略等，分明是在從無到有的設計一個產品。任何一個優秀的產品經理一定都應該明白這些道理。知其然并知其所以然。在分析設計策略中作者引到《竞争策略：行业与竞争者分析技术》的方法論，把產品設計和商業策略結合起來，事實上讀到這裡更加驗證了我之前一個觀點。在任何一個領域想要有所建樹，一定要有足夠廣闊的視野。就拿設計來說（雖然我不是設計）設計、美學理論是基礎是必修課，可是設計心裡學，色彩心理學，消費者行為心裡，認知心裡，產品設計策略一定是一個優秀的設計師所必須要知道的。只有知道了人們對一個東西感興趣的真正原因所在，才能設計出最好的東西。企業滿足用戶的需求，僱員滿足僱主的需求，設計師滿足產品經理的需求。只有當大家的需求都得到應有的滿足，社會才會變的和諧。嘻嘻

當然還有一個點，剛剛提到《竞争策略：行业与竞争者分析技术》這本書，雖然我還沒看，但是已經迫不及待了。國外這些非常牛逼的書之所以牛逼並不是因為把一個簡單的東西講的多麼高深，而是把一個高深的東西以一種平易近人的方式普及給大眾。這是我最喜歡看到的書了。回過頭來，話說一個前端工程師不好好研究前端，研究設計，研究UI，研究消費者行為，研究行業與競爭者分析技術幹什麼。其實，其實在我看來這就是前端的一部份，雖然這不是前端最主要的部份。無論是前端工程師還是後端工程師，都不應該只是寫代碼，寫代碼，寫代碼。更應該知道爲什麽產品要這麼設計，從用戶認知的角度，從商業競爭策略的角度。說到這裡不得不說阮一峰老師的博客，其實讀過阮一峰博客的人都大致有這樣一個體會，他的博客不是什麽陽春白雪的東西，幾乎都是科普性質的，適合各個知識層次的人，把一個新事物，用很通俗易懂的方式講述給大家。這也是作者的用意，讀者的需求。就想Denny那句話，任何一門知識，如果你不能給一個十歲的孩童講明白那證明你一定沒學會。

一个前端攻城湿眼的书法与设计美学

Author:Tengfei Date:2013-08-24 Categray: 设计

感谢Denny的建议，其实对书法对设计只是略知皮毛，但是有一些感悟想分享于大家。

书法从秦汉至今已有近2000年的历史，虽然从起源到现在书法在形式上经历了很多的变化，大小篆，草书，隶书，楷书，行书。但是她最最本质的审美却从未改变过。那，最本质的审美又是什么呢？用笔，结体。有人可能会问那章法呢？怎么没有章法？其实章法就是结体，只不过是放大了的结体。下面就两者分别分析一下。

用笔，也称笔法。我认为是书法里面最重要的。事实上我做过观察，发现很多很多的人对字的结构都把握的差不多，但是为什么写出来的字会有很大的差别呢，主要是用笔导致。例如同样找不同的几个人写“人”，“大”等字，它的结构已然很简单了，可是不同的用笔方法写出来的效果完全不一样。 说道用笔，不得不说中锋。中锋事实上是书法中的第一要法。如果一个字或者一副作品，脱离了中锋也就不能称之为书法了，因为无法可依了。当代著名女书法家孙晓云老师在她的《书法有法》中深刻细致的分析了中锋的意义及重要性。

还有如果从宏观上来说，书法就是“形”和“质”的组合，（是的，这两个字是我造出来的）。其实最重要的是“质”。“质”一句话说就是用笔的质量，点化的质量。说高手写字一个点就能看出功力，这是真的。中锋，起，行，收。这些都是书家极为讲究的。尤其是笔画给人感觉到的渗透力。若是“质”的境界到了，至于“形”却是千变万化的，每个人天生就和别人不一样，无论是写字画画唱歌性格。“形”就是结构，结体。其实无论是摄影，西方绘画，中国画。都比较重视这个的，被称之为构图。事实上构图无非就是用最基础的图元做扩充。中国画中的竹子，都是用“个”“介”这样的形法程式化的画出来的，但是高手又在这程式化的画里添加了自己的思想感情。中国画中的兰叶，都是用各种“撇”的上下组合，或急或缓，或曲或折。西方摄影和西方绘画以及中国画中画家们都利用了各种构图方式各种基础的几何元素，平行，各种样子的三角，菱形，平行，圆，椭圆。。。这样就显得有理有据。书法也利用了这些理论。或许古代的诸如《艺概》《书谱》这样的论著不会这样去说，但事实上都是用到了。几乎所有只要是和美有关的东西，包括平面设计，都会用到这些理论，当然我也觉得书法中可以用《写给大家看的设计书》中的那四个原则（亲密性，对齐，重复，对比）。

看到这些西方的美学理论的东西，顿时觉得古今中外，地球上的人这类动物对美有着惊人的相同的理解。

怎样维护一个开源js库

Author:Tengfei Date:2013-05-15 Categray: 思维
原文链接：https://segment.io/blog/tips-for-maintaining-an-open-source-library/ 作者：Calvin 翻译： 姚立 齐腾飞 马磊

5个月前，我们通过Hacker News发布了一个叫做Analytics.js的小类库。几个小时后，它的点击率就爬升到第一。在短短的一天内，项目的星（star）由20个变为超过1000个。从中我们学到了大量的管理开源类库的经验，因此，我想和你们分享其中的一些小窍门。

一开始，我们对管理开源类库一无所知。我认为，我们中的任何一个人在这之前都没有过合并pull requests代码的经验。因此，我们必须快速学习相关的知识。

现在，Analytics.js已经有超过2000个星了，开源社区中有很多人贡献了很棒的代码。一路走来，我们学到了很多让pull requests保持高质量的经验，以及如何简化贡献者的开发流程。

1. 保持编码风格的统一

新的开发者将会从现有的代码库中学习如何为库增添新的功能，而这才是他们应该做的事情。所有的开发者都想按照代码库的规范去增加新的功能而不是他们自己的规范。你的主要工作就是使这种规范尽可能的简单。新的开发者将会从现有的代码库中学习如何为库增添新的功能，而这才是他们应该做的事情。所有的开发者都想按照代码库的规范去增加新的功能而不是他们自己的规范。作为一个开源项目的维护着主要工作就是使以上情况尽可能的简单。

当你的分支代码和核心代码表意含糊不清时无数的麻烦就来了，如果你在两个不同的地方用两种不同的方式做同一件事情，开发者怎么知道哪一种方式才是推荐的呢？答案是：他们不知道。

更糟糕的是由于你的编码风格不统一，开发者无从决定，那么也会编码不统一（按自己的偏好来）。

解决这个问题你需要花费很多的时间规范代码风格。作为一个准则，你不应该尝试在一个简单的开源库中用两种不同的方式。如果你一定要改变风格，那一定要迅速的在全球都改变。否则新人将无法区分你之前代码和现有代码的约定。

我们一开始用一种很差劲的方式处理这个问题，所有的代码在一个文件里面，函数没有一个良好的结构组织。（并且如果你检查我们的提交，就必须先理清代码）我们在设计代码规范统一上没有花时间，所以库里的代码编写风格混杂不堪。

当pull代码的时候我们每个人都与别人的代码发生冲突，因为每个人都修改了同一个文件，并且在他们想要插入新函数的地方加入自己的功能函数

2. 使用正确的方法，简单的方法

最初我们构建代码的方式，导致在合并 pull requests 的时候遇到了许多问题：换句话说，刚开始我们的代码就没有结构。我们做出的改善代码结构的比较大的改变之一是使用 Component。

我们喜欢 Component 因为它消除了编码风格不统一的问题，减少了代码库的规模。Component 使用 CommonJS，所以我们只需要在我们需要的地方，**引入**（require）我们需要的模块即可。所有的事情都简单明了，这就意味着我们的代码对于新人来说更容易跟进。这是代码库维护者的梦想。

在进行切换期间，我们写了一堆自己的组件来替换已经附在全局对象**analytics**上的通用函数。现在，组件可以在库的任何地方很简单的包含进来和使用，pull requesters 只是默认使用他们。

自从我们发布了正确的方法，并且使他更加清晰，pull requests 的代码质量获得了显著提升。

为了尽可能保持统一编码风格的延续，当有新的代码提交上来的时候，你必须积极主动。不要害怕给 pull requests 的代码添加注释，即使看起来只是一处很小的更正，也不要害怕拒绝不必要的可能会搞乱你的 API 的 requests。

必须牢记的是，同样要严格要求自己的代码。如果在添加新特性时，自己开小差，懒得统一编码风格，那么为什么项目的贡献者不可以开小差呢？总之。你的代码越干净，对于新手来说就有越多的好的示例学习。

3. 写测试，并且连接到 Travis

写非常好的测试覆盖无疑是加速开发最简洁而有效的方法。由于我们 一直 在提交更改，所以我们不能因为担心现有功能遭到破坏而浪费时间。我们写了 很多 的测试，同时得到了很多的收获：更快的开发、对我们的代码更有信心、对贡献者的代码更有信心，还有很多……

这也会吸引更多高质量的 pull requests！

当开发者临摹了我们库的编码风格，会把它延伸到测试中去。在添加相应的测试之前，我们是不会接收贡献者的代码。

好的一面是我们不必过多的强制执行。感谢Travis-CI，几乎所有的 pull requests 是经过完全测试过的，而这些测试都是参考了我们**已有的测试案例**。既然我们坚信我们已有的测试有很好的质量，自然，由此而衍生来的测试便有了一个更高的起点。

拥有一个 passing Travis 标记也会增加信任，表明你的库文件正在正确的轨道上运行并仍然在维护。更不用说同行让你感到压力来保持你的测试在好的状态（显而易见这是最难的部分）。

4. 从第一天就开始版本管理

良好的版本管理，就像测试一样，需要约定规范并很值得这么去做。当开发者遇到问题时就会马上去检查运行的是不是最新的版本。如果不是，他们会更新代码并解决这个bug。如果没有版本管理，你仍然会收到提交的 bug 甚至是你早已解决了的bug。

我们开始的时候，都完全不知道如何控制版本；我们的代码仓库基本上只是一堆提交。如果你频繁的上传更新，就会导致使用你的代码库的开发人员不必要的困扰。

给代码仓库添加标记也被证明是非常有用的。每个版本都有一个定义良好且已经稳定的代码的包。如果你正使用一个前端打包工具比如 Component或者Bower，你也能得到非常好的自动打包。

5. 添加Makefile文件——黑客的指导手册

偷偷告诉你一个小秘密：我们几乎没有为那些想向这个类库贡献代码的人准备任何文档（这是另一个我们需要解决的问题）。但是我仍然很惊讶地发现，即使在没有任何构建或者测试手册的情况下，我们仍旧收到了很多的pull requests。

回过头来看下我们的代码库，那些新开发者是怎么知道如何构建脚本的呢？

正如我刚刚提到的，我们选择在Component上面构建流程。它远远领先我们老的构建流程，可惜并没有被广泛的接受（但是我们打赌终究会有那一天的）。

想象一下，大多数人之前并没有见过基于Component的代码库，我大胆猜测，他们是通过我们的Makefile文件学习的。make命令就是天然的入口点（译者注：make命令中的伪目标），从而了解整个新项目是怎样运行的。因此，它就成为既成事实的用户手册了。

我们在脚本中使用长标识（long flags），并且给每一个命令一个描述性的名称。因为使用Component构建我们的类库，我们可以首先参照他们的文档，然后再写自己的文档。一旦你理解了Component是如何工作的，你就会明白所有基于Component的代码库是如何构建的。

在不断 迭 代 代 码 的 过程中，我们做了很多工作来使我们的Makefile文件更加简洁明了。记住，你的构建和测试脚本也是你代码的一部分。它们也需要保持干净和可读性，因为它们是新来者最先接触到的东西。

6. 不断迭代流程

注意我所提到的所有技巧是如何精简流程的吗？这是因为维护一个流行项目必须如履薄冰，否则后果不堪设想。每当新的isuues出现时，你会不断地做小的修复工作，如果你不优化开发流程的话，你将会疲于奔波。

不仅如此，你的类库质量也会出现问题。如果没有一个好的构建系统、自动化测试和干净的代码库，修复小的bug就会像打扫家务一样繁琐，你需要花越来越多的时间去解决问题。当然没有人想要这样！

我们已经学到了很多管理像我们代码库这样大小的代码库的经验，但是面对管理一个真正大的项目时，我们还有很长的一段路需要走。管理一个大型的开源项目需要花费很多的功夫。随着代码量的增加，想要进行一次大的代码重构将会变得愈加困难。当越来越多的人依赖这个项目的时候，随之而来的pull requests也会开始增加。
